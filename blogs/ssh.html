ssh

Created2025-06-11|Updated2025-06-11

SSH（Secure Shell）是一种网络协议，用于在不安全的网络中提供安全的远程登录和其他网络服务。

ssh 基本工作流程

SSH 连接的建立分为两个阶段：密钥交换和会话建立。

  1. 密钥交换（Key Exchange）
    客户端发起连接：客户端向服务器发送连接请求，携带自己支持的 SSH 版本和加密算法列表。
    服务器响应：服务器选择一个共同支持的算法，并发送自己的主机公钥（存储在/etc/ssh/目录下）。
    密钥交换算法：双方使用 Diffie-Hellman 或 ECDH 等算法，在不安全的网络上协商出一个共享的会话密钥（Session Key）。
    该过程中，即使第三方截获通信，也无法计算出会话密钥。

  2. 身份验证（Authentication）
    服务器验证客户端的身份，常见方式有：
    密码验证：客户端发送用户名和密码（经会话密钥加密）。
    密钥对验证：客户端使用私钥对消息签名，服务器用客户端公钥验证（需提前将公钥添加到服务器的~/.ssh/authorized_keys）。
    证书验证：使用 CA 颁发的证书进行身份验证（企业环境常用）。

  3. 会话建立
    身份验证通过后，双方使用会话密钥加密所有后续通信。
    客户端和服务器进入交互式会话，所有命令和输出都被加密传输。

关于ssh中密钥验证

SSH 中的密钥验证是保障通信安全的核心机制，主要分为主机密钥验证和用户密钥验证两个部分

一、主机密钥验证（服务器身份验证）

  1. 验证目的
    确保客户端确实在与目标服务器通信，而非中间人攻击。
  2. 验证流程
    1. 服务器发送公钥
      客户端连接服务器时，服务器会发送自己的主机公钥（如 RSA、ED25519）。
    2. 客户端检查本地记录
      客户端在本地的 ~/.ssh/known_hosts 文件中查找该服务器的公钥记录：
    • 首次连接：无记录，客户端会提示用户确认公钥指纹（需要其他渠道验证 如密码验证）。
    • 非首次连接：比对新接收的公钥与记录是否一致。
    1. 验证结果处理
    • 匹配：连接继续，使用协商的会话密钥加密通信。
    • 不匹配：客户端发出警告（如 “REMOTE HOST IDENTIFICATION HAS CHANGED”），并拒绝连接。

二、用户密钥验证（客户端身份验证）

  1. 验证目的
    确认客户端用户是否拥有合法的私钥，替代传统密码登录。
  2. 验证流程
    1. 客户端生成密钥对
      客户端使用 ssh-keygen 生成密钥对（如 RSA）：
    2. 上传公钥到服务器
      客户端将公钥内容添加到服务器的 ~/.ssh/authorized_keys 文件中。
    3. 身份验证过程
    • 服务器向客户端发送一个随机挑战（Challenge）。
    • 客户端使用私钥对挑战进行签名，并将签名结果发送回服务器。
    • 服务器使用存储在 authorized_keys 中的公钥验证签名：
      • 验证通过：允许登录。
      • 验证失败：拒绝登录。
  3. 优势与应用
  • 安全性更高：避免密码明文传输，私钥难以破解。
  • 自动化场景：适合脚本或程序自动登录（如 CI/CD 部署）。
  • 多因素认证：可与其他认证方式（如密码、TOTP）结合使用。

三、常见问题与解决方案

  1. 主机密钥变更问题
  • 原因：服务器重装、密钥重新生成、中间人攻击。
  • 解决：
  1. 用户密钥验证失败
    排查步骤：
    1. 检查客户端私钥权限（应为 600）：chmod 600 ~/.ssh/id_rsa
    2. 确认服务器 authorized_keys 权限（600）和目录权限（700）。
    3. 查看服务器日志（如 /var/log/auth.log）获取详细错误信息。

关于主机密钥

SSH 主机密钥在 SSH 连接里起着关键作用，主要有以下两个方面：

  1. 验证服务器身份（核心功能）
    当你通过 SSH 连接到服务器时，比如 ssh user@example.com，服务器会把自己的公钥发送给客户端。客户端会对这个公钥进行验证，确认：
  • 该公钥是否和自己本地 ~/.ssh/known_hosts 文件中记录的一致。
  • 该公钥是否由可信赖的 CA（Certificate Authority）签名。

这样做能有效防止中间人攻击。打个比方，攻击者伪装成目标服务器，要是没有主机密钥验证，客户端就可能在不知情的情况下把数据发送给攻击者。

  1. 加密通信
    主机密钥是 SSH 建立安全通道的基础。在连接建立的初始阶段，客户端和服务器会借助主机密钥完成密钥交换，进而生成会话密钥。后续的数据传输都是通过这个会话密钥进行加密的。

密钥的生成与存储

  • 自动生成：使用 ssh-keygen -A 命令可以自动生成主机密钥，一般会生成 RSA、ECDSA、Ed25519 等不同类型的密钥。
  • 存储位置：这些密钥存放在服务器的 /etc/ssh/ 目录下，具体有：
    • /etc/ssh/ssh_host_rsa_key（私钥）
    • /etc/ssh/ssh_host_rsa_key.pub（公钥）
  • 客户端记录：客户端首次连接服务器后，服务器的公钥会被保存在用户家目录下的 ~/.ssh/known_hosts 文件中。

实际应用中的重要提示

  • 密钥变更提示：如果服务器的主机密钥发生了变化（像重新生成了密钥），客户端在下次连接时会发出警告，这其实是一种安全保护机制。
  • 定期更换密钥：为了提升安全性，建议定期更换主机密钥，更换后要通知所有客户端更新 known_hosts 文件。
  • 使用强密钥类型：优先选用 Ed25519 或 ECDSA 等现代密钥类型，避免使用 RSA-1024 这类较弱的密钥。

总结

主机密钥就像是服务器的“数字身份证”，它能确保：

  • 客户端确实是在和真正的目标服务器进行通信。
  • 通信双方的数据在传输过程中不会被窃取或篡改。
