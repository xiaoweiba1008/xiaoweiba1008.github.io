promise基本用法

JavaScript 中的 Promise 是一种处理异步操作的对象，用于表示一个异步操作的最终完成（或失败）及其结果值。它解决了传统回调地狱（Callback Hell）带来的代码可读性和维护性问题。

基本概念

  1. 三种状态：

    • pending：初始状态，既不是成功也不是失败。
    • fulfilled：操作成功完成。
    • rejected：操作失败。
    • 状态一旦改变（从 pending 变为 fulfilled 或 rejected），就会固定下来，不会再变。

  2. 链式调用：
    Promise 可以通过 .then() 和 .catch() 方法链式调用，避免嵌套过深。

  3. 错误处理：

    • .catch() 捕获 Promise 链中的任何错误。
    • 错误会 “冒泡” 到最近的 .catch()。

基本语法

javascript

关键方法

  1. **Promise.then()**：

    • 处理 Promise 的成功结果。
    • 可以返回一个新的 Promise 或值，用于链式调用。

  2. **Promise.catch()**：

    • 处理 Promise 的失败结果。
    • 等同于 .then(null, error => {})。

  3. **Promise.finally()**：

    • 无论 Promise 状态如何都会执行。
    • 不接收参数，常用于清理操作。

  4. 静态方法：

    • **Promise.all([...])**：并行处理多个 Promise，全部成功才返回成功，否则返回第一个失败的 Promise。
    • **Promise.race([...])**：返回最先完成（成功或失败）的 Promise。
    • **Promise.allSettled([...])**：返回所有 Promise 的结果（无论成功或失败）。
    • **Promise.any([...])**：返回第一个成功的 Promise，若全部失败则抛出 AggregateError。

常见错误

  1. 未处理的 Promise 拒绝：

    • 忘记添加 .catch() 可能导致错误被忽略。
    • 建议在 Promise 链末尾始终添加 .catch()。

  2. Promise 未返回：

    • 在链式调用中，如果某个 .then() 没有返回值或新的 Promise，可能导致后续 .then() 无法获取预期结果。

  • Promise 是处理异步操作的核心机制，避免了回调地狱。
  • 通过链式调用和错误冒泡，使代码更清晰、更易维护。
  • async/await 基于 Promise 构建，提供了更优雅的异步代码写法。

.then() 的返回值 链式中下一个 Promise 的状态
return value（普通值） 已解决（fulfilled），值为 value
return Promise.resolve(value) 已解决（fulfilled），值为 value

return 另一个 Promise 取决于这个 Promise 的最终状态

关键注意点

  1. 链式传递：.then() 的返回值会被包装成新的 Promise，传递给下一个 .then() 或 .catch()。
  2. 错误冒泡：一旦某个 .then() 中抛出错误或返回被拒绝的 Promise，后续的 .then() 会被跳过，直到遇到 .catch()。
  3. 异步与同步：无论返回值是同步值还是异步 Promise，后续处理都会等待当前 Promise 确定状态后再执行。

catch()方法 捕获  Promise 错误

.catch() 会捕获 整个 Promise 链中 任何位置抛出的错误，而不仅仅是最后一个 Promise 的错误。这是 Promise 链式调用的核心机制之一：错误会冒泡到最近的 .catch() 中被处理。

原理分析

  1. 错误冒泡机制：

    • 当 Promise 链中任意环节（如 fetch、response.text()、某个 then() 的回调）抛出错误或返回被拒绝的 Promise 时，后续的 then() 会被跳过，直接触发最近的 .catch()。
    • 无论错误发生在链的哪个位置，都会被传递到 .catch() 中。

  2. 就近捕获原则：

    • 如果有多个 .catch()，错误会被最近的上层 .catch() 捕获。
    • 未被捕获的错误会继续向上冒泡，直到被全局错误处理（如 window.onerror）捕获。

多个 .catch() 的情况

    • 如果 fetch 或第一个 then() 出错，被第一个 .catch() 捕获，链继续执行。
    • 如果第二个 then() 出错，被第二个 .catch() 捕获。

总结

  1. .catch() 捕获整个链的错误：

    • 无论错误发生在 fetch、response.text()、then() 回调还是其他环节，都会被传递到最近的 .catch()。
    • 错误会跳过后续的 then()，直接触发 .catch()。

  2. 最佳实践：

    • 在 Promise 链末尾始终添加 .catch()，确保所有潜在错误都能被处理。
    • 使用多个 .catch() 可以分段处理不同类型的错误（如网络错误、业务逻辑错误）。

  3. 与 try...catch 的类比：

    • Promise 链中的 .catch() 类似于同步代码中的 try...catch，但作用于异步操作。
    • 错误会从发生点 “冒泡” 到最近的错误处理器。

错误冒泡机制和原理

在 JavaScript 的 Promise 中，错误冒泡机制是指：当 Promise 链中的某个环节抛出错误或返回被拒绝的 Promise 时，错误会沿着 Promise 链向上传递（冒泡），直到被最近的 .catch() 或 try...catch 捕获处理。这一机制确保了异步操作中的错误可以被集中处理，避免 “幽灵错误”（未被捕获的错误导致程序崩溃）。

一、错误冒泡的核心原理

1. Promise 的状态与错误传递

  • Promise 有三种状态：pending（进行中）、fulfilled（已成功）、rejected（已失败）。
  • 当一个 Promise ** 被拒绝（rejected）** 时，它会将错误传递给后续的 消费环节（如 .then() 或 .catch()）。
  • 如果后续的 .then() 没有处理拒绝状态（即未提供第二个参数），错误会继续向后传递，直到遇到 .catch() 或链结束。

2. .then() 的双重角色

  • .then() 可以接收两个回调函数：

    • 第一个参数：处理 Promise ** 成功（fulfilled）** 的逻辑（onFulfilled）。
    • 第二个参数：处理 Promise ** 失败（rejected）** 的逻辑（onRejected）。

  • 如果未提供第二个参数，.then() 会将上游的拒绝状态（错误）原封不动地传递给下一个 Promise，相当于：

    javascript

二、错误冒泡的具体流程

假设存在以下 Promise 链：

javascript

流程解析：

  1. 初始 Promise 被拒绝：Promise.reject("初始错误") 生成一个状态为 rejected 的 Promise，携带错误 初始错误。
  2. **进入第一个 .then()**：
    • 未提供 onRejected 回调，因此错误不会被处理，直接传递给第一个 .then() 返回的新 Promise，使其状态变为 rejected，错误仍为 初始错误。
  3. **进入第二个 .then()**：
    • 同样未提供 onRejected 回调，错误继续传递，第二个 .then() 返回的新 Promise 状态仍为 rejected，错误仍为 初始错误。
  4. **遇到 .catch()**：
    • .catch() 本质上是 .then(undefined, onRejected) 的语法糖，专门处理拒绝状态。此时错误被捕获，执行 onRejected 回调（即 .catch() 中的逻辑）。

三、错误冒泡的关键规则

1. 错误会跳过后续的 .then()

  • 一旦某个 Promise 被拒绝，后续所有未处理拒绝状态的 .then() 都会被跳过，直接向链后方传递错误。

    javascript

2. 就近捕获原则

  • 错误会被离它最近的上层 .catch() 捕获，而不是链末尾的 .catch()。

    javascript

3. 错误可以在 .catch() 中被处理或重新抛出

  • 在 .catch() 中：

    • 处理错误并返回正常值：错误冒泡终止，后续 .then() 继续执行。
    • 重新抛出错误：错误会继续向链后方冒泡。

    javascript

在 Promise 链中，.catch() 相当于异步代码中的 catch 块，只不过错误冒泡是沿着 Promise 链的时序进行的：

五、典型错误场景与解决方案

场景 1：忘记处理 .then() 中的同步错误

javascript

  • 原理：两个 .then() 中抛出的错误都会被最后的 .catch() 捕获，因为它们没有自己的错误处理器。

场景 2：错误在 .catch() 中被吞掉

javascript

  • 风险：如果 .catch() 未处理或重新抛出错误，后续 .then() 会收到 undefined（若 .catch() 返回普通值）或忽略错误（若 .catch() 无返回）。

六、最佳实践

  1. **在 Promise 链末尾始终添加 .catch()**：
    • 确保任何未被处理的错误都能被捕获，避免程序崩溃。
  2. 分段处理错误：
    • 在需要的环节添加 .catch()，专门处理特定类型的错误（如网络错误、解析错误）。
  3. 避免在 .then() 中同时处理成功和失败：
    • 优先使用 .catch() 集中处理错误，保持代码逻辑清晰。

总结

  • 错误冒泡机制是 Promise 异步错误处理的核心，通过状态传递和就近捕获原则，确保错误可以被高效处理。
  • 理解这一机制的关键在于：**每个未处理拒绝状态的 .then() 都会将错误向后传递，直到遇到 .catch()**。
  • 合理使用 .catch() 可以避免异步代码中的 “错误黑洞”，提升程序的健壮性。

每次then() 方法返回一个新的 promise对象

在 Promise 链式调用中，每次 .then() 都会返回一个全新的 Promise 对象，而不是同一个对象。

一、每次 .then() 返回新的 Promise 对象

1. 核心规则

  • 每个 .then()/.catch() 都会返回一个新的 Promise，其状态取决于回调函数的执行结果：
    • 如果回调返回普通值，新 Promise 以该值成功（fulfilled）。
    • 如果回调抛出错误或返回被拒绝的 Promise，新 Promise 以该错误失败（rejected）。
    • 如果回调返回另一个 Promise，新 Promise 的状态取决于这个 Promise 的最终状态。

2. 示例验证

javascript

  • 结果：每个 .then() 返回的都是全新的 Promise 对象。

二、新 Promise 与原始 Promise 的关联

1. 状态传递机制

  • 上游 Promise 的状态决定下游 Promise 的执行：
    • 如果上游 Promise 成功，下游的 .then() 会执行 onFulfilled 回调。
    • 如果上游 Promise 失败，下游的 .catch() 会执行 onRejected 回调。

3. 错误传递

  • 上游 Promise 的错误会跳过后续的 .then()，直到遇到 .catch()：

三、错误冒泡与链式调用的关系

1. 错误冒泡的本质

  • 错误冒泡是链式调用的副作用：当某个 Promise 被拒绝时，错误会沿着链向后传递，直到被捕获。
  • 关键：错误不是在同一个 Promise 对象内部 “冒泡”，而是通过新 Promise 对象的状态传递实现的。

2. 示例分析

javascript

  • 流程解析：
    1. 第一个 .then() 抛出错误，返回 新的被拒绝的 Promise（记为 P1）。
    2. 第二个 .then() 接收 P1 的拒绝状态，由于未提供 onRejected，返回新的被拒绝的 Promise（记为 P2），错误保持不变。
    3. .catch() 接收 P2 的拒绝状态，执行 onRejected 回调。

四、链式调用的实现原理

1. Promise 链的本质

  • 每个 .then()/.catch() 返回的新 Promise 都会等待上游 Promise 确定状态后再执行回调，并根据回调结果决定自身状态。

  • 这一过程形成了链式依赖关系：

五、关键区别总结

特性 同一个 Promise 对象 链式调用的新 Promise 对象
状态是否可变 是（pending → fulfilled/rejected） 否（一旦创建，状态由上游决定）
能否重复使用 否（状态固定后无法改变） 是（每个新 Promise 独立）
错误处理方式 需在同一个 Promise 上添加回调 通过 .catch() 捕获上游错误
数据传递 单一值（成功或失败） 链式传递（上游值影响下游）

六、常见误区

误区 1：认为 .then() 是在原 Promise 上添加回调

  • 真相：.then() 不改变原 Promise，而是返回新 Promise。原 Promise 的状态一旦确定就不可变。

误区 2：认为错误冒泡是在同一个对象内部发生

  • 真相：错误是通过新 Promise 对象的状态传递实现的。每个新 Promise 都会根据上游状态决定是否执行回调。

误区 3：混淆 Promise 链与事件冒泡

  • 区别：事件冒泡发生在 DOM 树中，而 Promise 错误冒泡发生在 Promise 链的时序中。

总结

  1. 链式调用的本质：

    • 每个 .then()/.catch() 返回新 Promise，形成依赖链。
    • 上游 Promise 的状态决定下游回调的执行。

  2. 错误冒泡的本质：

    • 错误通过新 Promise 对象的状态传递，跳过后续 .then()，直到遇到 .catch()。
    • 错误冒泡是链式调用的自然结果，而非同一个对象内部的行为。

  3. 核心优势：

    • 链式调用使异步代码可组合、可复用，错误冒泡简化了错误处理逻辑。
